Introdução
Ref. ISO/IEC/IEEE 1016:2016 §5 — SDD content

Este documento descreve as decisões de design do sistema CRM Açaí Gurau, refinando a arquitetura definida no ADD-AG-001. Cobre o modelo de dados relacional, o contrato das principais rotas da API REST, os fluxos de integração WhatsApp e IA, a estrutura do frontend Vue.js e os padrões de código adotados. Serve como guia de implementação para o engenheiro responsável e como referência de manutenção futura.

activity_logs id, contact_id, user_id, type, description, meta_json, created_at Registro imutável de todas as ações
3.2 Índices e Performance
  • contacts.phone_whatsapp — índice único, usado para lookup ao receber mensagens
  • messages.contact_id + created_at — índice composto para histórico paginado
  • pipeline_items.stage_id + last_moved_at — usado para alertas de inatividade
  • activity_logs.contact_id — índice simples para timeline do contato

Fluxo de Integração WhatsApp + IA
Ref. ISO/IEC/IEEE 1016:2016 §5.6 — Dynamic design
5.1 Mensagem Recebida (fluxo principal)
1. Evolution API recebe mensagem WhatsApp do cliente B2B 2. Evolution API → POST /api/v1/webhooks/whatsapp (HMAC auth) 3. WhatsAppModule valida HMAC e cria Job: ProcessWhatsAppWebhook 4. Queue Worker executa Job: 4a. Busca contato pelo phone_whatsapp 4b. Se não encontrado → cria novo contato (lead automático) 4c. Salva mensagem na tabela messages (direction: in) 4d. Broadcast via Redis → frontend atualiza chat em tempo real 4e. Verifica ai_sessions: se status = active → cria Job: CallAIResponse 5. Job CallAIResponse: 5a. Busca contexto da conversa (últimas N mensagens + dados do contato) 5b. Monta prompt com AIProviderInterface::buildPrompt() 5c. Chama DeepSeek API (POST /v1/chat/completions) 5d. Salva resposta como message (direction: out, ai_generated: true) 5e. Envia via Evolution API → cliente recebe no WhatsApp
5.2 Handoff IA → Agente Humano
1. Agente clica em "Assumir atendimento" na interface 2. Frontend → POST /api/v1/contacts/{id}/ai/pause 3. AIModule atualiza ai_sessions.status = 'paused' 4. Jobs futuros de CallAIResponse verificam status e abortam 5. Novas mensagens chegam → apenas salvas e exibidas ao agente (sem resposta automática) 6. Agente pode reativar IA via POST /ai/resume a qualquer momento
5.3 Interface AIProviderInterface (troca de provedor)
interface AIProviderInterface { public function complete(array $messages, array $context): string; public function buildPrompt(Contact $contact, array $history): array; public function estimateTokens(array $messages): int; } // Implementações: class DeepSeekProvider implements AIProviderInterface { ... } class OpenAIProvider implements AIProviderInterface { ... } // futuro class ClaudeProvider implements AIProviderInterface { ... } // futuro // Binding no AppServiceProvider: $this->app->bind(AIProviderInterface::class, DeepSeekProvider::class);

6.2 Atualização em Tempo Real

O frontend usa Laravel Echo + Redis (via Pusher protocol) para receber eventos em tempo real sem polling. Ao receber uma nova mensagem WhatsApp, o backend dispara um broadcast no canal private-contact.{id}, que o frontend escuta e atualiza o chat automaticamente sem recarregar a página.

Design de Segurança
Ref. ISO/IEC/IEEE 1016:2016 §5.4 — RNF-SEG-* do SRS-AG-001
MecanismoImplementação
AutenticaçãoJWT com expiração de 8h. Refresh token com rotação (revogação após uso). Armazenados em HttpOnly cookies no frontend.
AutorizaçãoMiddleware de role no Laravel. Gates e Policies por recurso (ex: agente só acessa contatos atribuídos a ele).
Webhook HMACEvolution API assina cada webhook com HMAC-SHA256. crm-api valida assinatura antes de processar qualquer evento.
SQL InjectionTodas as queries via Eloquent ORM com bindings paramétricos. Nenhuma query raw sem sanitização.
SenhasHash bcrypt com custo 12 via Hash::make() do Laravel.
LGPDDados de terceiros (clientes B2B) não são enviados à DeepSeek API. O prompt inclui apenas tipo de segmento e histórico de conversa anonimizado. CNPJ e dados pessoais são omitidos do contexto de IA.
Rate Limiting60 req/min por IP nas rotas públicas. 300 req/min por usuário nas rotas autenticadas. Implementado via Laravel Throttle middleware.

Padrões de Implementação Adotados
Ref. ISO/IEC/IEEE 1016:2016 §5.7 — Design rationale
PadrãoAplicação no projeto
Repository PatternAbstração da camada de dados. Services consomem Repositories, não Models diretamente. Facilita testes unitários com mocks.
Provider/AdapterAIProviderInterface permite troca de provedor de IA sem impacto nos Services de negócio.
Command/JobOperações assíncronas (webhook processing, AI calls) implementadas como Laravel Jobs na fila Redis.
ObserverModel Observers para disparar activity_logs automaticamente em criação/atualização de contatos e pipeline items.
DTO (Data Transfer Object)Requests validados via Form Requests do Laravel. Respostas padronizadas via API Resources.
Composable (Vue.js)Lógica reutilizável extraída em composables: useWhatsApp, useAIPause, usePipelineDrag.
