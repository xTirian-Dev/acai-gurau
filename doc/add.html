Introdução
Ref. ISO/IEC/IEEE 42010:2011 §4 — Architecture description
1.1 Propósito

Este documento descreve a arquitetura do sistema CRM Açaí Gurau, identificando os principais elementos arquiteturais, suas responsabilidades, relacionamentos e as decisões que moldaram o design. Serve como referência para desenvolvimento, implantação e manutenção futura.

1.2 Sistema de Interesse (SoI)

O sistema de interesse é a plataforma CRM com automação WhatsApp e IA conversacional, composta por uma API REST em Laravel, um SPA em Vue.js, uma instância da Evolution API e integração com o serviço de IA DeepSeek V3.2. Todos os componentes rodam em containers Docker em um único VPS Hetzner.

7.3 Estratégia de Migração Futura

Caso o volume de mensagens WhatsApp comprometa os recursos do VPS principal, o container evolution-api pode ser movido para um VPS dedicado sem alterações no código. A única mudança necessária é atualizar a variável de ambiente EVOLUTION_API_URL na crm-api e ajustar as regras de rede.

8
Decisões Arquiteturais (ADRs)
Ref. ISO/IEC/IEEE 42010:2011 §5.7 — Architecture decisions
ADR-001 — VPS único com Docker Compose (vs. servidores separados)
StatusAceito
ContextoProjeto de baixo volume inicial (5 usuários), prazo de 2 meses, cliente sensível a custos de infraestrutura.
DecisãoTodos os containers em um único VPS Hetzner CX31 (~$20/mês), orquestrados por Docker Compose.
ConsequênciasRedução de custo de ~$9/mês (vs. VPS separado para Evolution API). Migração futura possível sem refatoração. Risco: contenção de recursos se volume de mensagens crescer muito.
ADR-002 — Evolution API self-hosted (vs. Z-API ou 360Dialog)
StatusAceito
ContextoNecessidade de integração WhatsApp sem custo de licença recorrente e com controle total sobre as sessões.
DecisãoEvolution API v2 open source, self-hosted no mesmo VPS via Docker.
ConsequênciasCusto zero de licença. Suporte a multi-sessão nativo. Risco: não é a API oficial do Meta (Baileys), sujeita a bloqueios de conta em uso abusivo. Mitigação: limitar volume e respeitar boas práticas de uso.
ADR-003 — DeepSeek V3.2 como provedor de IA inicial
StatusAceito
ContextoNecessidade de IA conversacional com custo mínimo para validação do produto no MVP.
DecisãoDeepSeek V3.2 ($0,28/M tokens input) como provedor inicial. Interface de IA encapsulada em AIModule com padrão Provider/Adapter para permitir troca futura.
ConsequênciasCusto estimado <$15/mês para o volume inicial. Troca para GPT-4o ou Claude possível sem refatoração dos módulos de negócio. Risco: latência ligeiramente maior que OpenAI em horários de pico.
ADR-004 — Arquitetura monolítica modular (vs. microserviços)
StatusAceito
ContextoEquipe de 1 desenvolvedor, prazo de 2 meses, baixo volume inicial.
DecisãoAPI Laravel única com módulos internos bem separados. Não microserviços.
ConsequênciasDesenvolvimento mais rápido, deploy mais simples. Refatoração para microserviços possível no futuro se escala exigir, graças à separação por módulos já adotada.
